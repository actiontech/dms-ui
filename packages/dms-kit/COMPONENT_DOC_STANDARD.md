# 组件文档优化标准

> 本文档用于指导组件 README.md 和 demo 的编写标准，确保文档简洁明了且适合 Cursor Rule 工具引用。

## 核心原则

### 1. 简洁明了
- 避免冗余信息和重复说明
- 聚焦核心功能和使用方法
- 用户视角：关注"是什么"、"怎么用"，而非实现细节

### 2. 避免重复
- **Demo 已展示的内容，文档不再重复**
- 状态管理、具体用法等在 demo 中完整展示即可
- 文档只需说明组件特点和注意事项
- "核心功能详解"不是必需的，只在确有必要时添加

### 3. 适合引用
- 结构清晰，分层明确
- API 说明准确，参数清晰
- 代码示例完整可用

## 文档结构标准

### 必需章节

```markdown
# 组件名称

一句话说明组件功能和核心价值（不超过 2 行）

## 何时使用

- 使用场景 1（简短描述）
- 使用场景 2
- 使用场景 3

（3-5 个核心场景，每个一行说明）

## 代码演示

### 基础用法

<code src="./demo/basic.tsx"></code>

### 其他场景（可选）

<code src="./demo/xxx.tsx"></code>

（demo 数量控制在 1-3 个，展示核心用法即可）

## API

| 参数 | 说明 | 类型 | 默认值 | 必填 |
| --- | --- | --- | --- | --- |
| prop1 | 简短说明 | `type` | - | ✅/- |

（必填项用 ✅ 标记，清晰直观）

## 组件特点

组件的核心功能和自动化能力说明：

1. **特点 1** → 说明
2. **特点 2** → 说明
3. **特点 3** → 说明

（3-5 个核心特点，突出组件价值）

## 核心功能详解（可选）

**何时需要：**
- ✅ 复杂组件，参数组合方式不明显
- ✅ 有多种使用模式，需要明确说明区别
- ✅ 关键功能需要代码示例才能理解
- ✅ Demo 无法充分展示所有用法

**何时不需要：**
- ❌ 简单组件，Demo 已完整展示用法
- ❌ API 表格已清晰说明所有参数
- ❌ 与 Demo 内容重复
- ❌ 只是重复 API 的文字描述

**编写原则：**
- 提供最小化代码片段，易于复制
- 解释"如何组合参数"实现特定功能
- 每个小节针对一个核心场景
- 代码示例要完整可运行

**示例：**
```markdown
## 核心功能详解

### 受控模式

通过 `editing` 外部控制编辑状态：

\`\`\`typescript
const [editing, setEditing] = useState(false);

<Component
  editing={editing}
  onStart={() => setEditing(true)}
  onEnd={() => setEditing(false)}
/>
\`\`\`

### 批量操作

使用 `items` 数组实现批量处理：

\`\`\`typescript
<Component
  items={[
    { id: 1, name: 'Item 1' },
    { id: 2, name: 'Item 2' }
  ]}
  onBatch={(selected) => handleBatch(selected)}
/>
\`\`\`
```

## 注意事项

1. 必须 xxx
2. xxx 需要处理 xxx
3. 父组件负责 xxx
4. xxx 仅在 xxx 时生效
5. 建议 xxx

（5-7 个关键注意点，帮助用户避坑）

## 最佳实践

1. **场景名称**：实践建议
2. **场景名称**：实践建议
3. **场景名称**：实践建议

（3-5 个实用建议）
```

### 禁止的章节

❌ **不要包含以下内容：**

- ~~设计规范~~（过于详细）
- ~~布局规范~~（实现细节）
- ~~主题配置~~（框架层面）
- ~~更新日志~~（维护在 CHANGELOG）
- ~~详细的使用示例~~（demo 已展示）
- ~~状态管理代码~~（实现细节）
- ~~样式特性~~（实现细节）

⚠️ **谨慎添加：**

- **核心功能详解**：对简单组件不需要，只在复杂组件确有必要时添加

## Demo 编写标准

### 文件命名

- `basic.tsx` - 基础用法（必需）
- `advanced.tsx` - 高级用法（可选）
- `场景名.tsx` - 特定场景（按需）

### 代码结构

```typescript
import React, { useState } from 'react';
import { 必要的 antd 组件 } from 'antd';
import { 组件名, ConfigProvider } from '@actiontech/dms-kit';

/**
 * demo 说明
 * - 特点 1
 * - 特点 2
 * - 特点 3
 */
const DemoName: React.FC = () => {
  // 状态管理
  const [state, setState] = useState(initialValue);

  // 事件处理
  const handleAction = () => {
    // 逻辑
  };

  return (
    <ConfigProvider>
      {/* demo 内容 */}
      <组件名
        prop1={value1}
        prop2={value2}
      />
    </ConfigProvider>
  );
};

export default DemoName;
```

### Demo 要求

1. **必须使用 ConfigProvider 包裹**
   ```typescript
   <ConfigProvider>
     <YourComponent />
   </ConfigProvider>
   ```

2. **添加顶部注释说明**
   ```typescript
   /**
    * 功能说明
    * - 特点 1
    * - 特点 2
    */
   ```

3. **代码简洁清晰**
   - 移除不必要的 Card 嵌套
   - 避免过多的辅助说明文字
   - 保持布局简洁

4. **展示核心场景**
   - 成功和失败情况都要展示
   - 不同配置的对比
   - 典型使用场景

5. **避免过度说明**
   - 不要在 demo 中添加大段文字说明
   - 不要展示状态值（除非必要）
   - 让代码本身说明用法

6. **按场景分离 demo 文件**
   - ⚠️ **不要把所有 demo 都集中在一个文件中**
   - 每个 demo 文件聚焦一个核心场景或功能
   - 保持每个 demo 的独立性和可读性
   - 便于用户快速找到需要的示例

## API 表格标准

### 必需列

| 参数 | 说明 | 类型 | 默认值 | 必填 |
| --- | --- | --- | --- | --- |

### 填写规范

1. **参数名**：使用代码格式
2. **说明**：简短直接，不超过 15 字
3. **类型**：使用反引号，如 `` `string` ``、`` `boolean` ``
4. **默认值**：没有填 `-`
5. **必填**：必填用 `✅`，非必填用 `-`

### 示例

```markdown
| 参数 | 说明 | 类型 | 默认值 | 必填 |
| --- | --- | --- | --- | --- |
| value | 当前值 | `string` | - | ✅ |
| onChange | 值变化回调 | `(value: string) => void` | - | ✅ |
| disabled | 是否禁用 | `boolean` | `false` | - |
```

## 文档长度原则

### 灵活控制

- 根据组件复杂度灵活调整文档长度
- 简单组件：精简扼要
- 复杂组件：详细说明，但避免冗余

### 精简方法

无论组件复杂度如何，都应：

1. 删除重复内容
2. 合并相似场景
3. 移除实现细节
4. 简化说明文字
5. demo 数量按需调整（核心是清晰展示功能）

### 判断标准

✅ **文档合适**：读者能快速理解组件功能和用法
❌ **文档过长**：包含过多实现细节或重复说明
